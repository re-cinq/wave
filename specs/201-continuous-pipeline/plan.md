# Implementation Plan: Continuous Pipeline Execution

**Branch**: `201-continuous-pipeline` | **Date**: 2026-03-16 | **Spec**: [#201](https://github.com/re-cinq/wave/issues/201)
**Input**: Feature specification from `/specs/201-continuous-pipeline/spec.md`

## Summary

Add a `--continuous` flag to `wave run` that wraps pipeline execution in a loop, automatically iterating over work items from a configurable source (GitHub issues, file queues). The loop controller lives in a new `internal/continuous/` package, delegating per-iteration execution to the existing `DefaultPipelineExecutor`. Graceful shutdown via context cancellation, structured observability events per iteration, and configurable failure policies (halt/skip).

## Technical Context

**Language/Version**: Go 1.25+
**Primary Dependencies**: `github.com/spf13/cobra` (CLI flags), existing `internal/pipeline`, `internal/event`, `internal/state` packages
**Storage**: Existing SQLite state store (no schema changes — each iteration creates a standard run record)
**Testing**: `go test ./...` with table-driven tests, race detector
**Target Platform**: Linux/macOS (single binary)
**Project Type**: Single Go project
**Performance Goals**: No measurable overhead on single-run execution (SC-005). Source polling is one-shot (fetch all items once, iterate locally).
**Constraints**: No new runtime dependencies. ContinuousRun is in-memory only (per spec C3). Max ~100 items per source fetch by default.
**Scale/Scope**: Batch processing up to 100 issues per continuous run. Each iteration is a full pipeline execution (~minutes per item).

## Implementation Phases

### Phase A: Core Loop Controller (`internal/continuous/`)

**Files**: `internal/continuous/runner.go`, `internal/continuous/runner_test.go`

1. Define `Runner` struct with fields: source, pipeline/manifest refs, executor factory, failure policy, max iterations, delay, emitter
2. Implement `Run(ctx context.Context) (*Summary, error)`:
   - Initialize `ContinuousRun` in-memory state
   - Loop: call `source.Next(ctx)` → check dedup → call executor factory to create fresh executor → `executor.Execute()` → record result → check failure policy → delay → check `ctx.Err()`
   - On SIGINT (context cancelled): break loop after current iteration completes
   - Return `Summary` with final counts
3. Implement `Summary` struct and human-readable `String()` method for FR-014
4. Tests: table-driven with mock executor and mock source covering:
   - Normal completion (all items succeed)
   - `on_failure: halt` stops on first failure
   - `on_failure: skip` continues past failures
   - `--max-iterations` cap
   - Context cancellation between iterations
   - Dedup: same item ID returned twice → skipped
   - Empty source → immediate clean exit

### Phase B: Work Item Sources (`internal/continuous/source.go`)

**Files**: `internal/continuous/source.go`, `internal/continuous/source_test.go`

1. Define `WorkItemSource` interface: `Next(ctx) (*WorkItem, error)`, `Name() string`
2. Implement `GitHubSource`:
   - On first `Next()`, shell out to `gh issue list --json number,url --label <label> --state <state> --limit <limit>`
   - Parse JSON response into `[]*WorkItem` (ID = issue number, Input = issue URL)
   - Subsequent `Next()` calls iterate through the pre-fetched list
   - Respect `sort` and `direction` params
3. Implement `FileSource`:
   - On construction, read all lines from file into `[]*WorkItem` (ID = line content hash, Input = line content)
   - `Next()` iterates through loaded items
4. Tests: mock `gh` output for GitHub source, temp files for file source

### Phase C: Source URI Parsing (`internal/continuous/parse.go`)

**Files**: `internal/continuous/parse.go`, `internal/continuous/parse_test.go`

1. `ParseSourceURI(uri string) (*SourceConfig, error)` — splits on first `:`, validates provider
2. `NewSourceFromConfig(cfg *SourceConfig) (WorkItemSource, error)` — factory dispatching to GitHubSource or FileSource
3. Validate known keys per provider, reject unknown providers
4. Tests: valid URIs, invalid providers, missing params, edge cases

### Phase D: Event Extension (`internal/event/emitter.go`)

**Files**: `internal/event/emitter.go`

1. Add fields to `Event` struct: `Iteration int`, `TotalProcessed int`, `WorkItemID string`
2. No changes to `EventEmitter` interface — these are optional fields on the existing `Emit()` method

### Phase E: CLI Integration (`cmd/wave/commands/run.go`)

**Files**: `cmd/wave/commands/run.go`

1. Add `Continuous`, `Source`, `MaxIterations`, `Delay`, `OnFailure` fields to `RunOptions`
2. Register flags on `wave run`: `--continuous`, `--source`, `--max-iterations`, `--delay`, `--on-failure`
3. Validate mutual exclusion: `--continuous` + `--from-step` → error (FR-010)
4. In `runRun()`: when `opts.Continuous` is true, parse source URI, construct `continuous.Runner`, call `runner.Run(ctx)`
5. Handle `Runner.Run()` result: print summary, set exit code (0 if all succeed, 1 if any failed per C4)
