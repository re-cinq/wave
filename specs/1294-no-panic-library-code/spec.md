## Context

From wave-scope-audit run `wave-scope-audit-20260422-223006-df0b`.

Several constructors and validators in library code still panic on bad input/environment conditions instead of returning errors. Three concrete offenders were called out by the audit: `NewStallWatchdog`, `NewGitHubClient`, and `ProvisionSkills`. Panics in library code are hostile to callers (CLI, webui, tests) because they cannot recover gracefully and cannot present meaningful errors to users.

Scope governance rule: *"No new panics in library code. Constructors and validators must return errors."*

Package-level notes reinforcing this:
- `internal/forge` verdict: *"return errors from constructors instead of panicking."*

## Lint Rule Decision

Use **golangci-lint `forbidigo`** (already part of `default: standard` linters set, configurable via the existing `.golangci.yml`). Pattern: `^panic$` scoped to non-test files inside `internal/`. Allowlist comment-based override (`//nolint:forbidigo // <reason>`) enforced via the existing `nolintlint` settings (`require-explanation: true`, `require-specific: true`).

This avoids a custom analyzer / `go vet` wrapper and keeps the policy in the same config the project already runs in CI.

- Converting `panic` calls in `cmd/`, test files, or generated code.
- Behavioural changes beyond signature changes and call-site adjustments.
