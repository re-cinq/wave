# Implementation Plan: Stacked Worktrees for Dependent Matrix Child Pipelines

**Branch**: `220-stacked-matrix-worktrees` | **Date**: 2026-03-16 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/220-stacked-matrix-worktrees/spec.md`

## Summary

Add a `stacked: true` option to matrix strategies that propagates branch context between dependency tiers. When enabled, each tier's child pipelines branch from the previous tier's output branch instead of the pipeline's default base. Multi-parent dependencies create temporary integration branches by merging parent branches. Existing non-stacked pipelines are unaffected.

## Implementation Phases

### Phase A: Type Extensions (FR-001, FR-006)

1. Add `Stacked bool` field to `MatrixStrategy` in `internal/pipeline/types.go:302`
2. Add `OutputBranch string` field to `MatrixResult` in `internal/pipeline/matrix.go:21`
3. Add `lastExecution *PipelineExecution` field to `DefaultPipelineExecutor`
4. Add `LastExecution() *PipelineExecution` method to `DefaultPipelineExecutor`
5. Set `e.lastExecution = execution` in `Execute()` method before returning

**Files**: `internal/pipeline/types.go`, `internal/pipeline/matrix.go`, `internal/pipeline/executor.go`

### Phase B: Output Branch Capture (FR-007)

1. Modify `childPipelineWorker` in `matrix.go:932` to capture the output branch from the child executor after `Execute()` returns
2. Use `childExecutor.LastExecution().WorktreePaths` to find the branch
3. Set `result.OutputBranch` to the first worktree branch found (child pipelines typically produce a single feature branch)

**Files**: `internal/pipeline/matrix.go`

### Phase C: Stacked Tier Execution (FR-002, FR-003, FR-008, FR-010)

1. Add `TierContext` struct to `matrix.go` to track `OutputBranches map[string]string`
2. In `tieredExecution()`, when `strategy.Stacked && strategy.DependencyKey != ""`:
   - Create a `TierContext` before the tier loop
   - After each tier completes, update `TierContext` with `result.OutputBranch` for each successful item
   - Before executing each item in a tier, resolve the base branch:
     - If item has a single dependency: use `TierContext.OutputBranches[depID]`
     - If item has no dependencies (tier 0): use default base
   - Pass the resolved base to the worker via a modified worker signature or by injecting base branch info into the execution context
3. When `stacked: true` but `dependency_key` is empty, fall through to the existing parallel execution (no-op per FR-008)
4. Emit progress events with stacking info per FR-010

**Files**: `internal/pipeline/matrix.go`

### Phase D: Multi-Parent Integration Branches (FR-004, FR-005, FR-009)

1. Add `createIntegrationBranch(pipelineID, itemID string, parentBranches []string) (string, error)` method to `MatrixExecutor`
2. Implementation:
   - Create a worktree from the first parent branch
   - Sequentially merge each additional parent branch
   - On merge conflict: return error with conflicting branch names (FR-005)
   - On success: return the integration branch name
3. Track integration branches in a `cleanupBranches []string` slice
4. Add `cleanupIntegrationBranches(branches []string)` that removes worktrees and deletes branches
5. Call cleanup after tier execution completes (FR-009), in a deferred function

**Files**: `internal/pipeline/matrix.go`, `internal/worktree/worktree.go` (if `Merge` helper needed, but likely just `exec.Command("git", "merge", ...)` directly)

### Phase E: Worker Base Branch Injection

1. The `matrixWorkerFunc` signature doesn't carry base branch info. Options:
   - **Option A**: Extend `matrixWorkerFunc` signature to include `baseBranch string` — breaks existing callers
   - **Option B**: Wrap the worker function in a closure that pre-configures the child pipeline's workspace base — cleaner
   - **Option C**: Inject base branch via the execution context
2. **Selected**: Option B — wrap `childPipelineWorker` to override the child pipeline's first worktree step base when stacking is active. The wrapper modifies the child pipeline's workspace config before passing to the child executor.

**Files**: `internal/pipeline/matrix.go`

### Phase F: Testing

1. **Unit tests for `TierContext`**: branch tracking across tiers
2. **Unit tests for integration branch creation**: single parent, multi-parent, merge conflict
3. **Integration test: 2-tier linear chain** (SC-001): tier 0 creates branch, tier 1 branches from it
4. **Integration test: diamond pattern** (SC-002): A → B,C → D
5. **Backward compatibility test** (SC-003): existing matrix tests pass unchanged
6. **Merge conflict test** (SC-004): conflicting branches produce actionable errors
7. **Integration branch cleanup test** (SC-005): branches removed after completion
8. **Stacked without dependency_key test** (FR-008): silently ignored

**Files**: `internal/pipeline/matrix_test.go`
