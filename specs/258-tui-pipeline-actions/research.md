# Research: TUI Finished Pipeline Interactions

**Date**: 2026-03-06
**Feature**: #258 — TUI Finished Pipeline Interactions

## R1: Chat Session — Direct Claude Code Invocation via tea.Exec()

**Decision**: Spawn `claude` binary directly as an interactive subprocess using `tea.Exec()`, bypassing Wave's adapter infrastructure entirely.

**Rationale**: Wave's adapter (`internal/adapter/claude.go`) is designed for non-interactive pipeline execution — structured prompts, contract validation, artifact I/O, and `--print` mode. An interactive chat session is a fundamentally different use case: the user wants a raw Claude Code REPL in a specific working directory. `tea.Exec()` is Bubble Tea's standard mechanism for suspending the program, handing terminal control to a subprocess, and resuming when the process exits.

**Key implementation points**:
- Construct `exec.Command("claude")` with `Dir` set to the workspace path
- Pass to `tea.Exec(cmd, callback)` where callback returns `ChatSessionEndedMsg{Err: err}`
- The subprocess inherits stdin/stdout/stderr automatically — Bubble Tea handles terminal state (alternate screen exit/enter, raw mode teardown/setup)
- After return, `ChatSessionEndedMsg` triggers a data refresh and git state refresh
- Validate workspace directory existence before launching (using `os.Stat()`)

**Alternatives Rejected**:
- Reuse adapter infrastructure: Designed for non-interactive pipeline execution; would require major refactoring and wouldn't provide interactive REPL behavior
- Direct `exec.Command` without `tea.Exec()`: Terminal state management becomes manual and error-prone
- `tea.Suspend`: Only pauses Bubble Tea without launching a process — user would need to manually run `claude` in another terminal

## R2: Workspace Path Reconstruction from RunRecord

**Decision**: Derive workspace path deterministically from `RunRecord.RunID` using the executor's known convention: `.wave/workspaces/<runID>/__wt_<sanitized_branch>/`, then validate existence with `os.Stat()`.

**Rationale**: The `StepStateRecord.WorkspacePath` column exists in the database schema but `SaveStepState()` always inserts `NULL` — workspace paths are only tracked in-memory during execution. The `RunRecord` stores `RunID` which IS the pipeline ID used as the workspace directory name (see `executor.go` line 286: `pipelineWsPath := filepath.Join(wsRoot, pipelineID)`). The worktree subdirectory uses `__wt_` + `sanitizeBranchName(branch)`.

**Key implementation points**:
- `RunRecord.RunID` is the pipeline execution ID (generated by `pipeline.GenerateRunID()` or `StateStore.CreateRun()`)
- Workspace root is `.wave/workspaces` (default from `executor.go`)
- For worktree workspaces: `.wave/workspaces/<RunID>/__wt_<sanitized_branch>/`
- The `sanitizeBranchName()` function is in `internal/pipeline/context.go` — replaces non-alphanumeric with `-`, collapses consecutive dashes, trims, limits to 50 chars
- Use `filepath.Glob(".wave/workspaces/<RunID>/__wt_*")` as fallback since the sanitized branch may not match perfectly
- Validate directory existence with `os.Stat()` — if not found, `WorkspacePath` is empty

**Alternatives Rejected**:
- Query `StepStateRecord.WorkspacePath` from DB: Always `NULL` in current implementation
- Store workspace path in `RunRecord`: Would require DB schema migration

## R3: Branch Existence Check in FetchFinishedDetail()

**Decision**: Run `git rev-parse --verify <branch>` during `FetchFinishedDetail()` to populate `BranchDeleted` on `FinishedDetail`.

**Rationale**: Branch existence must be checked somewhere. Checking on every cursor movement (in `emitSelectionMsg()`) would spawn a `git` process on every list navigation, causing lag. Checking during `FetchFinishedDetail()` (which already does I/O — state store queries) adds one more shell-out per detail fetch, which is acceptable since it happens once per selection.

**Key implementation points**:
- Run `exec.Command("git", "rev-parse", "--verify", branch).CombinedOutput()` with error checking
- Non-zero exit code means branch doesn't exist → `BranchDeleted = true`
- The `PipelineSelectedMsg.BranchDeleted` field remains unused for now — the detail model determines branch status from its own `FinishedDetail` data
- This approach aligns with C11's recommendation

**Alternatives Rejected**:
- Check on every cursor movement: Performance concern — spawning `git` on every keystroke
- Check only when action key is pressed: User wouldn't see fainted hints until they try the action
- Use `go-git` library: Adds an external dependency for a simple shell-out

## R4: Branch Checkout — Background git Command

**Decision**: Execute `git checkout <branch>` as a background `exec.Command()` (not `tea.Exec()`), returning result as a `BranchCheckoutMsg`.

**Rationale**: Unlike the chat session and diff view, branch checkout is non-interactive — it runs in the background and reports success/failure. Using `tea.Exec()` would unnecessarily suspend the TUI for a sub-second operation. A `tea.Cmd` function runs the checkout and returns a message with the result.

**Key implementation points**:
- Run `exec.Command("git", "checkout", branch).CombinedOutput()` in a `tea.Cmd` closure
- Before checkout, validate branch exists via `git rev-parse --verify <branch>`
- Return `BranchCheckoutMsg{Success: bool, Err: error}`
- On success: trigger `GitRefreshTickMsg` for header bar to detect the new branch
- On failure: display transient error in the action hints area

**Alternatives Rejected**:
- `tea.Exec()` with `git checkout`: Unnecessary TUI suspension for a non-interactive command
- Direct `os/exec` without message: Violates Bubble Tea's event-driven architecture

## R5: Diff View — External Pager via tea.Exec()

**Decision**: Use `tea.Exec()` to run `git diff main...<branch>` which pipes through the user's configured `core.pager`.

**Rationale**: Git automatically pipes `diff` output through the configured pager (default: `less`). Using `tea.Exec()` suspends the TUI, gives the pager full terminal control, and resumes cleanly on exit. This respects user preferences (many developers configure `delta` or `diff-so-fancy`) and handles large diffs naturally.

**Key implementation points**:
- Construct `exec.Command("git", "diff", "main..."+branch)` — the triple-dot syntax computes the merge-base
- Git handles pager invocation automatically based on `core.pager` config
- The `DiffViewEndedMsg` callback triggers TUI resume
- No explicit `Dir` needed — process CWD is the project root (where `wave tui` was launched)

**Alternatives Rejected**:
- Inline diff rendering in viewport: Would need to reimplement diff colorization, doesn't respect user's pager preference
- `git diff main..<branch>` (two dots): Shows symmetric difference, not just the branch's changes since divergence
- Detecting default branch name: Unnecessary complexity — Wave uses `main` conventionally, and `git diff` shows a clear error if it doesn't exist

## R6: Transient Error Display in Action Hints Area

**Decision**: Display action errors as a styled message in the right pane, replacing the action hints temporarily. Errors clear on next key press or navigation.

**Rationale**: The spec (FR-014, C8) requires transient error display. Rather than replacing the entire detail view (which the `stateError` pattern does), errors should overlay the action hints area at the bottom of the finished detail view. This preserves context — the user can still see pipeline summary while understanding why the action failed.

**Key implementation points**:
- Add `actionError string` field to `PipelineDetailModel`
- When `actionError` is non-empty, `renderFinishedDetail()` renders the error in red instead of action hints
- Clear `actionError` on any subsequent `tea.KeyMsg` or `PipelineSelectedMsg`
- Error styling: red foreground using lipgloss Color("1")

**Alternatives Rejected**:
- Full `stateError` transition: Loses the detail context, requires extra Esc to navigate back
- Toast/notification system: Over-engineered for simple error display

## R7: Status Bar — FinishedDetailActiveMsg Priority

**Decision**: Add `FinishedDetailActiveMsg{Active bool}` following the `FormActiveMsg` and `LiveOutputActiveMsg` pattern, with priority after `liveOutputActive`.

**Rationale**: The status bar hint chain is: `formActive` → `liveOutputActive` → `finishedDetailActive` → generic right-pane → left-pane. This mirrors the existing pattern established by #256 (FormActiveMsg) and #257 (LiveOutputActiveMsg). The finished detail hints show `"[Enter] Chat  [b] Branch  [d] Diff  [Esc] Back"`.

**Key implementation points**:
- Add `finishedDetailActive bool` field to `StatusBarModel`
- Handle `FinishedDetailActiveMsg` in `StatusBarModel.Update()`
- Forward `FinishedDetailActiveMsg` to status bar in `AppModel.Update()` (same pattern as `FormActiveMsg`, `LiveOutputActiveMsg`)
- Emit `FinishedDetailActiveMsg{Active: true}` when entering `stateFinishedDetail` with right pane focused
- Emit `FinishedDetailActiveMsg{Active: false}` when leaving via Esc or state change

**Alternatives Rejected**:
- Single `DetailStateMsg` carrying state enum: More complex, less explicit than boolean Active flag
- Status bar polling content state: Violates message-passing architecture

## R8: `[d]` Diff Hint Fainting Consistency (C15)

**Decision**: Both `[b]` and `[d]` hints must be fainted when `branchDeleted` is true or `BranchName` is empty.

**Rationale**: The existing `renderFinishedDetail()` code (from prior PRs) faints `[b]` when `branchDeleted` but does NOT faint `[d]`. Since diff also depends on branch existence, both must be treated consistently. The `[Enter]` chat hint is NOT fainted by branch deletion — chat depends on workspace path, which may exist even if the branch was force-deleted.

**Key implementation points**:
- In `renderFinishedDetail()`, also apply `Faint(true)` to `diffHint` when `branchDeleted` is true
- Add `BranchName` empty check alongside `branchDeleted` check for both `[b]` and `[d]`
- The key handler gating in `Update()` mirrors the rendering: `b` and `d` disabled when branch is deleted/empty
