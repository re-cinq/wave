# Implementation Plan: Init Merge & Upgrade Workflow

**Branch**: `230-init-merge-upgrade` | **Date**: 2026-03-04 | **Spec**: [specs/230-init-merge-upgrade/spec.md](spec.md)
**Input**: Feature specification from `/specs/230-init-merge-upgrade/spec.md`

## Summary

Enhance `wave init --merge` to display a categorized change summary before modifying files, add manifest diff preview, ensure consistent `--force`/`--merge`/`--yes` flag semantics, and add post-upgrade migration verification guidance. The implementation extends the existing init command in `cmd/wave/commands/init.go` with a pre-mutation computation phase, file content comparison against `go:embed` defaults, and a confirmation prompt. Integration tests cover the full upgrade lifecycle.

## Implementation Approach

### Phase A: Core Types and Change Summary Computation

1. **Add types** to `init.go`: `FileStatus`, `FileChangeEntry`, `ManifestAction`, `ManifestChangeEntry`, `ChangeSummary` (see data-model.md)

2. **Add `computeChangeSummary()`**: Compares each asset file against embedded defaults using `bytes.Equal`. Returns a `ChangeSummary` struct with file-level and manifest-level changes.

3. **Add `computeManifestDiff()`**: Modified version of `mergeManifests()` that tracks what keys were added vs preserved during the deep merge. Returns both the merged manifest and a list of `ManifestChangeEntry`.

### Phase B: Display and Confirmation

4. **Add `displayChangeSummary()`**: Renders the change summary as a formatted table to stderr. Uses ANSI colors if terminal supports it. Groups by category (personas, pipelines, contracts, prompts, manifest).

5. **Add `confirmMerge()`**: Prompts for user confirmation unless `--yes` or `--force` is specified. In non-interactive terminals, requires `--yes`/`--force` or aborts with helpful message.

### Phase C: Refactor runMerge()

6. **Refactor `runMerge()`**: Replace the current immediate-write flow with the pre-mutation pattern:
   - Parse existing manifest (abort with clear error on parse failure per FR-013)
   - Call `computeChangeSummary()` to build the summary
   - If `AlreadyUpToDate`, print "Already up to date" and exit (per US1-AS4)
   - Call `displayChangeSummary()` to show the summary
   - Call `confirmMerge()` for confirmation
   - Call `applyChanges()` to write only confirmed changes

7. **Add `applyChanges()`**: Writes the merged manifest and creates only "new" files. Handles filesystem permission errors per edge case 3.

### Phase D: Flag Semantics

8. **Update flag interaction logic**: Ensure `--merge --force` and `--merge --yes` both skip confirmation but still display summary to stderr (FR-007, FR-008). Ensure `--force` without `--merge` overwrites all files (FR-009). Handle non-interactive detection (FR-014).

### Phase E: Migration Guidance

9. **Update `printMergeSuccess()`**: Add `wave migrate up` to the "Next steps" section (C-4).

10. **Update `wave migrate status` output**: The existing `newMigrateStatusCmd()` already reports pending migrations (FR-012). Verify it handles missing `.wave/state.db` gracefully (edge case 5 — `NewMigrationRunner` already creates the DB).

### Phase F: Tests

11. **Add change summary unit tests**: Test `computeChangeSummary()` with various file states (new, preserved, up-to-date).

12. **Add manifest diff tests**: Test `computeManifestDiff()` with nested keys, arrays, additions.

13. **Add integration lifecycle test** (`TestInitMergeUpgradeLifecycle`): init → customize → init --merge → verify summary → verify file preservation → verify additions.

14. **Add flag combination tests**: All four combinations per FR-009/FR-010.

15. **Add edge case tests**: Malformed YAML, empty persona file, read-only .wave/, non-interactive terminal.

### Phase G: Documentation

16. **Write `docs/upgrade-guide.md`**: Step-by-step upgrade workflow documentation (FR-016).
