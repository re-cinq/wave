# Implementation Plan: Hierarchical Skill Configuration

**Branch**: `385-skill-hierarchy-config` | **Date**: 2026-03-14 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/385-skill-hierarchy-config/spec.md`

## Summary

Add a `skills:` field at three configuration scopes — global (`wave.yaml` top-level),
per-persona, and per-pipeline — with merge precedence pipeline > persona > global.
All skill references are validated against the `.wave/skills/` DirectoryStore at load
time. Resolution happens per-step at execution time since each step may use a different
persona. The resolved skill set drives SKILL.md provisioning into workspaces.

## Implementation Phases

### Phase A: Data Model & YAML Parsing (FR-001, FR-002, FR-003, FR-010)

Add `Skills []string` fields to `Manifest`, `Persona`, and `Pipeline` structs.

**Files**:
- `internal/manifest/types.go` — Add `Skills []string \`yaml:"skills,omitempty"\`` to `Manifest` and `Persona`
- `internal/pipeline/types.go` — Add `Skills []string \`yaml:"skills,omitempty"\`` to `Pipeline`

**Validation**: Existing YAML unmarshal handles absent/null/empty via `omitempty` tag.
Test with round-trip marshal/unmarshal (SC-001).

### Phase B: Skill Resolution Function (FR-004, FR-005, SC-002, SC-005)

Create `internal/skill/resolve.go` with `ResolveSkills(global, persona, pipeline []string) []string`.

**Algorithm**:
1. Create `seen map[string]bool`
2. Iterate pipeline → persona → global (precedence order for future config-carrying skills)
3. For each name not in `seen`, add to result and mark seen
4. Sort result alphabetically
5. Return

**Tests** (`resolve_test.go`):
- Empty inputs → empty output
- Single scope → sorted output
- All three scopes → merged, deduplicated, sorted
- Duplicate across scopes → appears once
- Same input order → same output (determinism)
- Pipeline + `requires.skills` keys unified before call

### Phase C: Skill Validation (FR-006, FR-007, FR-011, SC-003, SC-006)

Create `internal/skill/validate.go` with `ValidateSkillRefs(names []string, scope string, store Store) []error`.

**Logic**:
1. For each name in `names`:
   a. Call `ValidateName(name)` — format check
   b. If format valid and `store != nil`, call `store.Read(name)` — existence check
   c. On error, append to error slice with scope annotation
2. Return aggregated errors

**Integration points**:
- `manifest.ValidateWithFile()` — call `ValidateSkillRefs` for `m.Skills` and each `persona.Skills`
- `pipeline.ValidateDAG()` or new validation path — call `ValidateSkillRefs` for `p.Skills`

**Signature change for manifest validation**: `ValidateWithFile` needs access to a
`skill.Store`. Options:
- Add `store skill.Store` parameter to `ValidateWithFile` (breaking change, acceptable in prototype)
- Create a new `ValidateWithStore` function that wraps `ValidateWithFile`
- Use a `ValidateOptions` struct

**Decision**: Add a new function `ValidateManifestSkills(m *Manifest, store skill.Store) []error`
called separately from the existing `ValidateWithFile`. This avoids breaking existing callers
and keeps skill validation clearly separated. The `Load()` function calls both.

**Pipeline validation**: Add a `ValidatePipelineSkills(p *Pipeline, store skill.Store) []error`
function called from the executor or the pipeline loading path.

### Phase D: Executor Integration (FR-012, FR-013, FR-009)

Modify `executor.go` to:

1. **Resolve skills per-step** in `buildAdapterRunConfig()`:
   - Get global skills from `execution.Manifest.Skills`
   - Get persona skills from `execution.Manifest.GetPersona(step.Persona).Skills`
   - Get pipeline skills from `execution.Pipeline.Skills` + `execution.Pipeline.Requires.SkillNames()`
   - Call `skill.ResolveSkills(global, persona, pipeline)`

2. **Provision from DirectoryStore** for name-only references:
   - Existing `Provisioner` handles `SkillConfig`-backed skills
   - For name-only references (in resolved set but not in `requires.skills`), use
     `DirectoryStore.Read(name)` to get `Skill.Body` and write it into the workspace
   - Write SKILL.md content to `.wave/skills/<name>/SKILL.md` in the workspace
   - Copy resource files using `Skill.ResourcePaths`

3. **Backward compatibility** (FR-008, SC-004):
   - Existing `requires.skills` flow unchanged
   - Empty `Skills` fields at all scopes → existing behavior

### Phase E: Testing & Validation (SC-001 through SC-006)
