# Implementation Plan: Publish Wave Skills as Standalone SKILL.md Artifacts

**Branch**: `559-skills-publish` | **Date**: 2026-03-24 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/559-skills-publish/spec.md`

## Summary

Implement `wave skills audit`, `wave skills publish`, and `wave skills verify` subcommands that enable classification, publishing, and integrity verification of Wave skills. Skills are classified using content-based heuristic analysis (Wave-reference counting), published via the Tessl CLI, content-addressed with SHA-256 digests, and tracked in an atomic lockfile at `.wave/skills.lock`. A `skills/` directory at repo root holds the distributable standalone skill set.

## Technical Context

**Language/Version**: Go 1.25+
**Primary Dependencies**: `github.com/spf13/cobra`, `gopkg.in/yaml.v3`, `crypto/sha256`, `encoding/json`
**Storage**: Filesystem (`.wave/skills.lock` JSON lockfile)
**Testing**: `go test ./...` with table-driven tests
**Target Platform**: Linux, macOS (single static binary)
**Project Type**: Single CLI binary
**Performance Goals**: Single skill publish < 10s excluding network latency (SC-007)
**Constraints**: Must delegate to `tessl` CLI for registry operations; no new runtime dependencies
**Scale/Scope**: 13 built-in skills, extensible to user-installed skills

## Implementation Phases

### Phase A: Core Library (internal/skill/)

**A1: Skill Classification** (`classify.go`, `classify_test.go`)
- `ClassifySkill(skill Skill) SkillClassification` — scan body for Wave-specific keywords, count references, assign tag
- `ClassifyAll(store Store) ([]SkillClassification, error)` — classify all skills from store
- Keywords: `wave`, `.wave/`, `pipeline`, `persona`, `wave.yaml`, `manifest`, `worktree`, `wave run`, `wave init`
- Thresholds: 0 → `standalone`, 1-10 → `both`, >10 → `wave-specific`
- Must read full skill body (not just metadata) via `store.Read(name)` for classification

**A2: Content Digest** (`digest.go`, `digest_test.go`)
- `ComputeDigest(skill Skill) (string, error)` — SHA-256 over SKILL.md bytes + sorted resource files
- Format: `sha256:<hex>` prefix for future algorithm extensibility
- Must re-read files from `skill.SourcePath` to get raw bytes (not parsed content)
- Resource files sorted alphabetically by relative path for deterministic ordering

**A3: Lockfile Management** (`lockfile.go`, `lockfile_test.go`)
- `Lockfile` struct with `Version int` and `Published []PublishRecord`
- `LoadLockfile(path)` — parse JSON, handle missing file (return empty Lockfile), handle corrupt JSON (return error)
- `Save(path)` — atomic write via temp file + `os.Rename()`
- `FindByName(name)` — lookup for idempotency check (FR-007)
- `Upsert(record)` — insert or replace by name for post-publish updates

**A4: Publish Validation** (extend `validate.go`)
- `ValidateForPublish(skill Skill) ValidationReport` — validate required fields (errors) + optional fields (warnings)
- Required (error): `name` (format-checked), `description`
- Optional (warning): missing `license`, missing `compatibility`, missing `allowed-tools`
- `ValidationReport.Valid()` returns true if no errors (warnings allowed)

**A5: Publish Workflow** (`publish.go`, `publish_test.go`)
- `Publisher` struct: `store Store`, `lockfilePath string`, `registryName string`, `lookPath lookPathFunc`
- `PublishOne(ctx, name string, opts PublishOpts) PublishResult` workflow:
  1. Read skill from store
  2. Validate via `ValidateForPublish()` → reject if invalid
  3. Classify via `ClassifySkill()` → warn if wave-specific (unless opts.Force)
  4. Compute digest via `ComputeDigest()`
  5. Check lockfile for existing entry with same digest → skip if unchanged (FR-007)
  6. If not dry-run: call `tessl publish <skillDir>` via `exec.CommandContext`
  7. Update lockfile with new `PublishRecord`
- `PublishAll(ctx, opts PublishOpts) []PublishResult` — list + classify + filter eligible + iterate `PublishOne()`
- `PublishOpts`: `Force bool`, `DryRun bool`, `Registry string`

### Phase B: CLI Commands (cmd/wave/commands/)

**B1: Error Codes** (extend `errors.go`)
- `CodeSkillPublishFailed = "skill_publish_failed"`
- `CodeSkillValidationFailed = "skill_validation_failed"`
- `CodeSkillAlreadyExists = "skill_already_exists"`

**B2: Audit Subcommand** (extend `skills.go`)
- `wave skills audit [--format json|table]`
- Reuses `newSkillStore()` → reads full skills → `ClassifyAll()` → render
- Table columns: NAME | CLASSIFICATION | WAVE REFS | WARNINGS | SOURCE
- Summary line: "X standalone, Y wave-specific, Z both (N total)"
- JSON: `SkillAuditOutput` struct (see data-model.md)

**B3: Publish Subcommand** (extend `skills.go`)
- `wave skills publish <name> [--all] [--force] [--dry-run] [--registry <name>] [--format json|table]`
- Single skill: `<name>` arg required (unless `--all`)
- `--all`: batch publish all eligible skills
- `--force`: override wave-specific warning
- `--dry-run`: validate + digest only, no publish
- `--registry`: target registry (default: "tessl")
- Table output: per-skill status line (✓ Published / ⊘ Skipped / ✗ Failed)
- JSON: `SkillPublishOutput` struct

**B4: Verify Subcommand** (extend `skills.go`)
- `wave skills verify [--format json|table]`
- Load lockfile → for each published entry → re-read skill → compute digest → compare
- Table columns: NAME | STATUS | EXPECTED | ACTUAL
- Summary: "X ok, Y modified, Z missing"
- JSON: `SkillVerifyOutput` struct

### Phase C: Distributable Skills Directory

**C1: Populate `skills/` directory**
- Copy all `standalone` + `both` classified skills from `.claude/skills/` to `skills/`
- Exclude `wave` skill (wave-specific, 139 wave refs)
- Include resource files (scripts/, references/, assets/) for each copied skill
- Include a `README.md` in `skills/` explaining the directory purpose (only if requested)

### Phase D: Integration & Testing

**D1: Integration tests** — end-to-end audit/publish/verify with mock tessl binary
**D2: Lockfile round-trip tests** — write/read/corrupt/recover scenarios
**D3: Digest determinism tests** — same input produces same digest across runs
**D4: Classification accuracy tests** — verify all 13 skills classified correctly

## Key Design Decisions

1. **Tessl CLI delegation**: Publishing via `tessl publish <path>` subprocess call, consistent with existing `tessl search` and `tessl install` patterns in `source_cli.go`. No direct HTTP API calls.

2. **Content-based classification**: Automated Wave-reference counting instead of manual frontmatter annotation. Thresholds tuned to current 13-skill corpus. Deterministic and maintainable.

3. **Atomic lockfile**: Write-to-temp-then-rename prevents partial state on failure (FR-012). Standard POSIX pattern.

4. **SHA-256 digest format**: `sha256:<hex>` prefix allows future algorithm migration without breaking existing lockfiles.

5. **Extend existing files**: Validation extends `validate.go`, CLI commands extend `skills.go`, error codes extend `errors.go`. Follows established project patterns, avoids file proliferation.

6. **`skills/` at repo root**: Distributable directory separate from `.claude/skills/` (which contains wave-specific skills). Only publish-eligible skills are included (FR-016).

7. **Mock tessl for tests**: Integration tests use a mock `tessl` binary (shell script in temp dir) to avoid real registry calls, following the `lookPathFunc` injection pattern from `source_cli.go`.
