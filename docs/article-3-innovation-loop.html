01 Guilford's Insight: Two Modes of Thinking

In 1956, psychologist J.P. Guilford made a distinction that would reshape creativity research: convergent vs. divergent thinking.

üåä
Divergent Thinking
Generate multiple novel ideas by exploring many possible solutions. Spontaneous, free-flowing, non-linear.
  • Brainstorming sessions
  • Exploring alternative approaches
  • Challenging assumptions
  • "What if we tried...?"
üéØ
Convergent Thinking
Find the single best solution by applying logic, rules, and established knowledge. Systematic, structured, linear.
  • Implementing known patterns
  • Following specifications
  • Optimizing existing solutions
  • "This is how we do it"

"Divergent thinking involves more creativity. Convergent thinking is the opposite‚Äîit organizes and structures ideas, following logical steps to arrive at one solution."

‚Äî Wikipedia, synthesizing Guilford's research

The key insight: these aren't personality types‚Äîthey're cognitive modes. Everyone uses both. Effective problem-solving requires oscillating between them.

In software development, this maps to:

  • Divergent: Prototyping, architecture exploration, research spikes, brainstorming solutions
  • Convergent: Bug fixes, refactoring, implementing well-understood features, optimization

Most AI coding methodologies optimize for one mode. TDD, BDD, spec-driven development‚Äîall convergent. "Just vibe with the AI" is divergent. Neither alone is sufficient.

02 The Double Diamond in Code

The British Design Council's "Double Diamond" visualizes how divergent and convergent thinking alternate in creative work. Applied to software:

The Double Diamond in Software Development

Diamond 1 (Problem Space): First diverge to discover all aspects of the problem. Then converge on a clear problem definition. "What are we actually solving?"

Diamond 2 (Solution Space): Diverge again to explore potential solutions. Then converge to implement the chosen approach. "How do we solve it?"

Notice the rhythm: expand ‚Üí narrow ‚Üí expand ‚Üí narrow. Each phase has different cognitive demands‚Äîand different tool requirements.

03 Cursor's Multi-Agent Lessons

Cursor's recent research on long-running autonomous coding provides empirical validation. Running hundreds of agents for weeks, they discovered that role separation maps directly to cognitive modes.

üî¨ From Cursor's Research

"With no hierarchy, agents became risk-averse. They avoided difficult tasks and made small, safe changes instead. No agent took responsibility for hard problems or end-to-end implementation."

Their solution: separate Planners from Workers.

  • Planners (divergent mode): Continuously explore the codebase, create tasks, spawn sub-planners for specific areas. They ask "what could we do?"
  • Workers (convergent mode): Pick up tasks, focus entirely on completing them, don't coordinate with other workers. They ask "how do we do this specific thing?"

This solved coordination problems and let them scale to very large projects. The key insight:

"The right amount of structure is somewhere in the middle. Too little structure and agents conflict, duplicate work, and drift. Too much structure creates fragility."

‚Äî Cursor Blog, "Scaling Long-Running Autonomous Coding"

Translated: divergent phases need freedom; convergent phases need constraints. Mixing them creates chaos.

04 When SDD Works (and When It Fails)

Spec-driven development (SDD) emerged as a way to constrain AI agents. The hypothesis: comprehensive upfront specifications reduce hallucinations and improve quality.

In production, patterns emerge:

Scenario Use SDD? Why
Greenfield with clear requirements ‚úì Yes Specs provide structure; requirements stable
Regulated domains (finance, healthcare) ‚úì Yes Documentation is deliverable, not overhead
Large teams, cross-boundary coordination ‚úì Yes Specs create alignment across groups
Large existing codebase ‚úó No Context blindness; can't spec what you don't know
Exploratory development / prototypes ‚úó No Premature specification; requirements emerging
Small bug fixes (1-3 files) ‚úó No Overhead exceeds benefit; sledgehammer for nut
Rapidly evolving requirements ‚úó No Spec maintenance burden exceeds value

The fundamental issue: SDD front-loads everything. It assumes you know what you need upfront and that agents will follow comprehensive instructions. Production shows both assumptions are fragile.

The Double Review Problem

Specs often contain code snippets as examples. These require review before implementation. Then the generated code requires review again‚Äîagents inevitably deviate or encounter edge cases not covered.

Review time doubles rather than decreasing. The spec becomes a second codebase requiring maintenance.

06 Graph Navigation Serves Both Modes

The temporal graphs from Article 2 support both cognitive modes‚Äîthey're infrastructure, not methodology.

For Convergent Work

When implementing "add 2FA following our existing auth patterns," temporal graphs reveal those patterns immediately. The agent sees that auth-middleware.ts and session-manager.ts are modified together 47 times. It loads both, recognizes the established pattern, and implements consistently.

This is efficient convergence‚Äîfollowing known paths to known solutions.

For Divergent Work

When exploring "how could we improve our auth system?", the temporal graph surfaces unexpected connections. Co-modification analysis shows rate-limiter.ts is frequently changed with auth-middleware.ts (28 times).

You weren't thinking about rate limiting. But the graph reveals it as a related concern, triggering questions:

  • Why are these coupled?
  • Should 2FA integrate with rate limiting?
  • What other security features cluster with auth?
  • Are there patterns we're missing?

The graph doesn't answer these‚Äîit surfaces them. Divergent mode means following these leads to discover insights.

07 No Silver Bullet‚ÄîBetter Architecture

Fred Brooks was right in 1986. There is no silver bullet for software complexity. LLMs don't change this fundamental nature‚Äîthey're powerful tools for managing accidental complexity, not eliminating essential complexity.

But they introduce new accidental complexities:

  • Context window limitations and autocompact pressure
  • Hallucinations and non-deterministic behavior
  • Token economics and cost management
  • Security boundaries in autonomous systems

These aren't essential difficulties‚Äîthey're artifacts of our current tools. They can be addressed through better architecture.

The Innovation Loop

Oscillate deliberately between divergent (exploration) and convergent (exploitation) modes. SDD for stable requirements, natural language for discovery. Graphs serve both.

The real innovation isn't any single technique. It's recognizing that AI-assisted development requires rethinking our information architecture:

  • From: Tools designed for human navigation patterns ‚Üí To: Tools designed for probabilistic relevance ranking
  • From: Documentation written for human reference ‚Üí To: Context structured for progressive disclosure
  • From: Permissions designed for human actors ‚Üí To: Permissions designed for autonomous workspace isolation
  • From: One methodology for all problems ‚Üí To: Deliberate oscillation between cognitive modes

Graph-augmented navigation is one piece. Skills are another. Permission boundaries are a third. The innovation loop is a fourth.

We're in the middle innings. Tools evolving rapidly, methodologies still being discovered, best practices emerging through production experience.

But the direction is clear: selective context loading informed by how developers actually work, not just how code is currently structured. And deliberate switching between exploration and exploitation, not defaulting to one mode.

üîÑ Key Takeaway

Lead with divergence, follow with convergence. Use graphs to reveal when to switch. No silver bullet‚Äîbut better architecture makes everything else work.
